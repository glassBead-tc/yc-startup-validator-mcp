# Michael Seibel Startup Evaluation Framework

## Core Philosophy

Michael Seibel's evaluation approach is built on **systematic product building**, **data-driven iteration**, and **rapid execution**. He emphasizes structured processes over ad-hoc development, measurable KPIs over vanity metrics, and continuous user feedback loops over theoretical planning. His framework focuses on building MVPs quickly, establishing clear metrics, and using the scientific method to iterate toward product-market fit.

**Central Principle:** "If your process revolves around arguing, not writing spec, and long dev cycles, you are doing it wrong."

## The Scientific Method for Startups

### The Core Process: Hypothesize > A/B Test > Conclude > Repeat

#### 1. Data-Driven Decision Making
- **Measurement requirement**: Analytics must be in place from minute one
- **Question accessibility**: Everyone should be able to ask questions and get answers quickly
- **Hypothesis testing**: Every decision should be backed by data
- **Continuous experimentation**: Constant testing and iteration

#### 2. Analytics-First Approach
- **Immediate integration**: Add analytics to new features from day one
- **Universal access**: Use analytics tools that non-programmers can use
- **Question volume**: The easier to ask questions, the more questions you'll ask
- **Employee usage**: Track how often team members use analytics

### The Scientific Method Evaluation:
- **Question**: "How do you measure and test your assumptions?"
- **Good answer**: Specific analytics tools, testing processes, and measurement frameworks
- **Bad answer**: "Obvious" decisions without measurement or gut-feel approaches
- **Michael's insight**: "The harder it is to ask a question, the fewer questions you will ask"

## The MVP Development Framework

### MVP Philosophy: Launch Quickly, Then Iterate

#### 1. Speed Over Perfection
- **Quick launch**: Get something in users' hands as fast as possible
- **Iteration focus**: Improve based on real user feedback
- **Learning priority**: MVPs are for learning, not perfection
- **User testing**: Use MVP to understand what users actually want

#### 2. The Hair-on-Fire Problem Test
- **Problem urgency**: Are users desperate for this solution?
- **Immediate need**: Do users need this right now?
- **Pain intensity**: How badly does this problem hurt?
- **Solution validation**: Does the MVP actually solve the problem?

### MVP Evaluation Criteria:

#### Lean MVP Examples:
- **Airbnb**: Simple website with basic listings
- **Stripe**: Developer-focused payment API
- **Twitch**: Basic live streaming platform

#### Heavy MVP Warning Signs:
- **Months of development**: Long development cycles before launch
- **Feature completeness**: Waiting for "full" product before launching
- **Perfectionism**: Obsessing over polish instead of functionality
- **No user feedback**: Building in isolation without user input

### The MVP Questions:
- "What's the simplest version that solves the core problem?"
- "How quickly can you get this in front of users?"
- "What will you learn from the first version?"
- "How will you measure success?"

## The Product Development Cycle Framework

### The 1-Week Sprint System

#### Step 1: Set KPI Goals
- **Primary KPI**: Usually revenue (or usage if never charging)
- **Company-wide visibility**: Everyone knows the number every day
- **Measurement clarity**: Clear, unambiguous metrics
- **Business relevance**: KPIs that reflect company health

#### Step 2: KPI-Focused Brainstorming
- **Cycle theme**: Choose one KPI to improve this cycle
- **Inclusive process**: Everyone contributes ideas
- **Visual organization**: Write all ideas on whiteboard
- **Psychological safety**: All ideas are considered and recorded

#### Step 3: Easy/Medium/Hard Classification
- **Objective standards**: Clear criteria for effort required
- **Hard**: Takes one engineer most of the dev cycle
- **Medium**: 1-2 days of work
- **Easy**: Multiple can be done in a day
- **Reframing**: Most hard ideas can be restated as easy ideas

#### Step 4: Prioritization Process
- **Hard first**: Evaluate high-impact difficult features first
- **Impact assessment**: Which will move KPIs the most?
- **Effort vs. impact**: Balance difficulty against potential results
- **Team buy-in**: Process removes ego from decision-making

#### Step 5: Written Specifications
- **Clear documentation**: Detailed specs for all selected features
- **Task assignment**: Specific ownership of each component
- **Timeline clarity**: Clear deadlines and deliverables
- **Testing integration**: Built-in testing and measurement plans

### The Development Cycle Evaluation:
- **Question**: "What's your product development process?"
- **Good answer**: Structured cycle with clear KPIs, measurement, and iteration
- **Bad answer**: Ad-hoc development without clear process or metrics
- **Red flags**: Long cycles, arguing without data, no written specs

## The KPI and Measurement Framework

### KPI Selection Criteria:

#### Primary KPI Guidelines:
- **Revenue first**: Almost always should be revenue
- **Usage alternative**: DAU/MAU only if never charging users
- **Business health**: Reflects how good the company is doing
- **Actionable**: Can be influenced by team actions

#### Supporting KPIs:
- **Contributor metrics**: KPIs that drive the primary KPI
- **Measurable impact**: Clear relationship to main metric
- **Team focus**: Each cycle focuses on one supporting KPI
- **Improvement tracking**: Week-over-week progress monitoring

### The Justin.tv/Twitch KPI Structure:
- **Primary KPI**: Daily Active Users (DAU)
- **Supporting KPIs**: 
  1. New users (acquisition)
  2. Retention (engagement)
  3. New content created (supply)

### Measurement Best Practices:

#### Analytics Implementation:
- **Standard tools**: Use established analytics platforms
- **Team accessibility**: Non-technical team members can query data
- **Real-time insights**: Quick answers to business questions
- **Unlimited questions**: No barriers to asking questions

#### Testing Framework:
- **A/B testing**: Compare different approaches systematically
- **Hypothesis formation**: Clear predictions before testing
- **Statistical significance**: Proper test design and analysis
- **Iteration speed**: Rapid testing and learning cycles

## The Systematic Building Approach

### Process Over Chaos:

#### 1. Structured Meetings
- **Single team meeting**: One product meeting per cycle
- **Clear agenda**: Focus on specific KPI improvement
- **Time-boxed**: Efficient use of team time
- **Documented decisions**: Written records of all choices

#### 2. Inclusive Decision-Making
- **Everyone contributes**: All team members participate in brainstorming
- **Idea validation**: All ideas are recorded and considered
- **Objective evaluation**: Data-driven decision criteria
- **Ego removal**: Process reduces personal conflicts

#### 3. Execution Discipline
- **"Shut up and get to work"**: Focus on building after planning
- **Spec adherence**: Build according to written specifications
- **Testing integration**: Continuous testing throughout development
- **Delivery commitment**: Meet cycle deadlines consistently

### The Process Evaluation:
- **Question**: "How do you run your product development process?"
- **Good answer**: Structured cycles, clear KPIs, inclusive brainstorming, written specs
- **Bad answer**: Endless meetings, arguing without data, no clear process
- **Michael's test**: "Does your process create clarity or confusion?"

## User Feedback Integration

### Systematic User Research:

#### 1. Personal Problem Connection
- **Founder experience**: Solve problems you've personally experienced
- **Domain expertise**: Deep understanding of user needs
- **Authentic motivation**: Genuine passion for solving the problem
- **User empathy**: Real connection to target users

#### 2. User Testing Process
- **Early engagement**: Test with users before and after launch
- **Continuous feedback**: Regular user contact throughout development
- **Behavior observation**: Watch what users do, not just what they say
- **Iteration trigger**: User feedback drives product changes

### The User Research Framework:
- **Question**: "How do you validate your ideas with users?"
- **Good answer**: Systematic user testing, regular feedback collection, behavior analysis
- **Bad answer**: Assumptions without validation, building in isolation
- **Process requirement**: Regular user contact and feedback integration

## Red Flags in Michael's Framework

### Development Process Red Flags:
1. **Long development cycles**: Months without user feedback
2. **Arguing without data**: Decisions based on opinions not facts
3. **No written specs**: Unclear development requirements
4. **Perfectionism**: Waiting for "complete" product before launch
5. **No measurement**: Building without analytics or KPIs

### Team Process Red Flags:
1. **Exclusive decision-making**: Founders making all decisions alone
2. **Idea dismissal**: Shooting down ideas without consideration
3. **Ego-driven debates**: Personal conflicts over product decisions
4. **No structured meetings**: Meandering discussions without outcomes
5. **Lack of ownership**: No clear responsibility for deliverables

### Product Development Red Flags:
1. **No MVP approach**: Trying to build fully-formed products
2. **Feature creep**: Adding features without user validation
3. **No user feedback**: Building in isolation
4. **Vanity metrics**: Tracking metrics that don't reflect business health
5. **No testing**: Making changes without A/B testing

## The Investment Decision Framework

### High-Conviction Investment Criteria:
- **Systematic approach**: Clear, structured development process
- **Data-driven decisions**: Analytics and measurement in place
- **Rapid iteration**: Quick build-test-learn cycles
- **User-focused building**: Strong user feedback integration
- **Clear KPIs**: Measurable business metrics and goals

### Pass Criteria:
- **Chaotic development**: No clear process or structure
- **Opinion-based decisions**: Lack of data or measurement
- **Perfectionism**: Long cycles without user feedback
- **Founder-only decisions**: Exclusive, non-transparent process
- **Vanity metrics**: Tracking meaningless numbers

## The Systematic Success Formula

### Michael's Core Framework:
"Have a process to get product out the door. Establish clear KPIs. Create structured brainstorming. Use easy/medium/hard classification. Write specs. Shut up and get to work. Test everything."

### Success Indicators:
- **Structured development**: Clear, repeatable process
- **Measurable progress**: KPI-driven improvement
- **Rapid iteration**: Quick build-test-learn cycles
- **Team alignment**: Everyone understands goals and process
- **User validation**: Continuous feedback integration

This framework represents Michael Seibel's authentic approach to startup evaluation, emphasizing systematic product building, data-driven iteration, and rapid execution over chaotic development.
